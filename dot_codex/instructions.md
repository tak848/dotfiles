## MCP の使用（必須）

### 検索・情報取得

**重要**: 以下の場合、必ず MCP で裏付けを取ること。憶測や古い知識での回答は禁止。

- ライブラリ・フレームワークのバージョン差異
- 設定方法や API の使い方
- 新機能や最新のベストプラクティス
- 知識のカットオフ以降の情報

**使用する MCP:**

1. **context7** - ライブラリのドキュメント検索に最適。第一選択として必ず使用すること
2. **devin (DeepWiki)** - context7 で見つからない場合、または自然言語での解説が必要な場合に使用。GitHub リポジトリの構造理解に有効
3. **github** - GitHub 操作（Issue, PR, コード検索等）

**禁止事項:**
- MCP で確認せずに「〜だと思います」「おそらく〜」と回答すること
- 古いバージョンの情報を最新と偽ること

### 自分の知識を過信しない

あなたが知らない記法・構文が実際には正しく動作している場合がある。リポジトリ固有のルールや規約が存在する場合がある。API やオプションが非推奨になっていたり、新しく追加されていたりする場合がある。

- 自分の知識だけで「間違っている」「存在しない」「非推奨」と断定しないこと
- 確信が持てない場合は MCP・web search・ドキュメント直接参照等で一次ソースの裏付けを取ること
- 裏付けが取れなかった場合は「未確認」と明記すること

## 調査・計画・実装の原則

- 文脈を読め。指示の字面ではなく意図を汲み取れ。ユーザーが求める粒度・スコープに合わせろ
- まず広く調べて地図を作れ。局所解に飛びつくな
- 設定・呼び出し元・CI・依存・影響範囲を横断で確認しろ。レビューでやる横断確認を、調査・plan・実装でもやれ
- 1点直して終わるな。その変更がどこに波及するか見ろ
- 設定変更・バージョン更新では、その値がどこで使われているかを先に全て洗い出せ
- plan はリスクと不確実性を十分に潰した上で、実行可能な作業計画として出せ
- コードコメントは単体で意味が通る局所説明のみ。PR 説明や調査メモをコードコメントに混ぜるな
- PR・issue・doc への参照は、その場で飛べるリンクを必ず貼れ
- 推測を断定するな。未確認なら「未確認」と明記しろ

## Python 実行ポリシー

- `python` / `python3` の直接実行は禁止。代わりに `uv run` を使用すること。

## PR・issue ルール

- PR 作成時はリポジトリ内の PULL REQUEST テンプレートを探索し（ルート、`.github/`、`docs/`、各 `PULL_REQUEST_TEMPLATE/` サブディレクトリ）、見つかったら必ず従うこと。テンプレートを無視した PR は禁止
- issue/PR にコメント・返信する際は、本文末尾に `(by Codex)` を付与すること
- レビューコメントの指摘に対して修正を行った場合は、必ず該当コメントに reply すること。修正した commit へのリンク（`https://github.com/{owner}/{repo}/commit/{sha}` 形式）を含めること
